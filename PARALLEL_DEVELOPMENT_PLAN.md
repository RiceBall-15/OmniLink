# OmniLink 全速并行开发计划

## 开发策略

**核心理念：** 最大化并行度，快速推进后端业务逻辑，同时完善前端功能

**并行度：** 5 个独立任务同时进行
**资源分配：** 充分利用 delegate_task 的多任务能力

---

## 任务分组

### 🚀 组 1：后端核心服务（并行 3 任务）

#### 任务 1.1：im-api - REST API 核心功能
**目标：** 实现用户认证、消息、会话等核心 API

**具体功能：**
- 用户注册/登录 API
- JWT Token 验证中间件
- 消息 CRUD API
- 会话管理 API
- 文件上传 API（基础）

**技术要点：**
- 使用 Actix-web 或 Axum 框架
- 数据库操作（PostgreSQL）
- Redis 缓存集成
- 请求验证和错误处理

**预期输出：**
- 完整的 API 路由实现
- 数据库迁移脚本
- API 文档（OpenAPI/Swagger）
- 单元测试（至少 10 个测试用例）

---

#### 任务 1.2：user-service - 用户认证和管理
**目标：** 实现完整的用户管理功能

**具体功能：**
- 用户注册/登录逻辑
- 密码加密（bcrypt）
- JWT Token 生成和验证
- 用户资料管理
- 权限系统（RBAC）
- 在线状态管理

**技术要点：**
- PostgreSQL 用户表设计
- Redis Session 管理
- 邮箱验证（可选）
- 密码重置（可选）

**预期输出：**
- 用户服务完整实现
- 数据库 Schema
- JWT Token 工具库
- 权限中间件
- 单元测试（至少 15 个测试用例）

---

#### 任务 1.3：im-gateway - WebSocket 网关完善
**目标：** 完善 WebSocket 连接管理和消息路由

**具体功能：**
- WebSocket 连接管理
- 心跳保活机制
- 消息路由（单播/广播）
- 连接认证
- 在线状态同步
- 断线重连处理

**技术要点：**
- 使用 tokio-tungstenite
- 消息队列（Kafka）
- Redis Pub/Sub
- 连接池管理

**预期输出：**
- WebSocket 网关完整实现
- 心跳保活机制
- 消息路由逻辑
- 连接认证中间件
- 单元测试（至少 8 个测试用例）

---

### 🎨 组 2：前端增强功能（并行 2 任务）

#### 任务 2.1：消息编辑/撤回功能
**目标：** 实现消息的编辑和撤回功能

**具体功能：**
- 消息编辑（双击或右键菜单）
- 消息撤回（2分钟内）
- 撤回提示（"XXX 撤回了一条消息"）
- 编辑历史记录
- 权限控制（只能编辑/撤回自己的消息）

**技术要点：**
- 添加编辑状态管理
- 实现撤回倒计时
- WebSocket 消息同步
- UI 优化（编辑模式、撤回样式）

**预期输出：**
- 消息编辑组件
- 消息撤回组件
- 相关 Hooks 和 Services
- 单元测试（至少 5 个测试用例）

---

#### 任务 2.2：消息已读回执 + 在线状态
**目标：** 实现消息已读状态和用户在线状态同步

**具体功能：**
- 消息已读回执
- 已读状态显示（√ / √√）
- 在线状态显示（在线/离线/忙碌）
- 在线用户列表
- 实时状态同步

**技术要点：**
- WebSocket 状态同步
- 消息已读标记
- 在线状态轮询/推送
- 状态管理优化

**预期输出：**
- 消息已读组件
- 在线状态组件
- 状态同步逻辑
- 单元测试（至少 8 个测试用例）

## 风险评估

### 高风险
- ⚠️ 服务器资源不足，可能无法同时运行所有服务
  - **缓解：** 使用 Docker Compose 的资源限制，按需启动服务

### 中风险
- ⚠️ 并行任务可能产生代码冲突
  - **缓解：** 明确服务边界，使用独立代码库

### 低风险
- ⚠️ 测试覆盖不足
  - **缓解：** 强制要求单元测试，代码审查

## 紧急预案

**如果服务器资源严重不足：**
1. 暂停部分非核心服务
2. 优先完成核心功能
3. 使用轻量级替代方案

**如果某个任务阻塞：**
1. 立即切换到其他并行任务
2. 记录阻塞问题，稍后解决
3. 不影响其他任务进度
